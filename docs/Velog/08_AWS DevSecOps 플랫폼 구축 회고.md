# 설계대로 된 건 없었다 — AWS DevSecOps 플랫폼 구축 회고

개인 프로젝트를 시작한 이유는 단순히 보안 기능 하나를 구현해보고 싶어서가 아니었다.
그룹 프로젝트에서는 보안 파트를 주로 담당했지만, 쿠버네티스, CI/CD, 인프라 구성 같은 영역은 팀원들이 맡았다. 프로젝트가 끝난 뒤 내가 직접 전체 흐름을 설계하고 끝까지 운영해봐야겠다는 생각이 들었다. 이 프로젝트는 그 부족한 부분을 채우기 위한 시도였다.

---

### 권한은 문서대로 되지 않는다

WAF 로그를 S3에 적재하는 것부터 막혔다. 코드는 문제없었지만 로그가 쌓이지 않았다. KMS, WAF, S3 세 서비스 사이의 권한이 맞지 않은 것이 원인이었다. 이후에도 비슷한 이유로 로그가 다시 멈췄다. 같은 종류의 실수를 반복했다는 게 더 남는 교훈이었다. 처음부터 원리를 이해하고 넘겼다면 두 번째는 없었을 것이다.

---

### $30 청구서가 날아온 날

KMS API 호출이 하루 614만 건, 청구가 $30을 넘었다. Cost Explorer에서 이상을 발견하고 CloudTrail을 역추적해서 Lambda 무한루프가 원인임을 찾아냈다. 공유 KMS 키로 전환하고 S3 Bucket Key를 활성화해서 해결했다. 비용 문제는 교과서에서 읽는 것과 직접 맞닥뜨리는 것이 완전히 다르다.

---

### 계획이 바뀌며 구조가 나아졌다

처음엔 Prometheus로 AI 탐지 결과를 Grafana Cloud에 실시간 연결하려 했다. 로컬 WSL의 `localhost`는 외부 클라우드에서 접근이 안 된다는 걸 직접 해보고 나서 알았다. 방향을 바꿔 S3 → Athena → Grafana 기반 준실시간 구조로 전환했고, 덕분에 SOAR 파이프라인까지 자연스럽게 연결됐다. 처음 계획대로 됐으면 오히려 더 단순한 구조에 머물렀을 것이다.

---

### 피처 하나가 모델을 바꿨다

Isolation Forest 성능이 기대보다 낮았다. WAF 로그가 URI path와 query string을 분리 저장하는데, 전처리에서 query string을 뭉개고 있었다. SQLi 페이로드는 query string에 집중되는데 엔트로피를 따로 측정하지 않으니 탐지율이 낮을 수밖에 없었다. `uri_entropy` 하나를 `path_entropy`와 `args_entropy`로 분리하자 Recall이 100%로 올랐다. 데이터를 이해하는 것이 모델 선택보다 먼저라는 걸 다시 확인했다.

---

### 마치며

처음 목표는 보안 외의 영역을 경험해보는 것이었다. CI/CD를 직접 구축했고, Terraform으로 인프라를 코드화했고, Kubernetes에 서비스를 올렸다. 그 과정에서 비용 폭탄도 맞고, 권한 문제로 막히고, 아키텍처를 뒤집는 결정도 했다.

돌아보면 잘 만든 구조보다 많이 부딪혔다는 게 더 기억에 남는다. 설계대로 흘러간 것보다 예상 밖에서 막힌 지점들이 결국 더 많이 배운 순간들이었다.
