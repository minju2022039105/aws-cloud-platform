## [2026-05-07] ALB 기반 구조에서 Serverless(API GW + Lambda)로의 완전 전환 및 트러블슈팅 
---

### 1. 작업 개요 및 진행 상황
*   **리소스 정리:** 구형 리소스(ALB, EC2, 이전 WAF)의 의존성 분석 완료 및 4단계 강제 삭제 성공.
*   **코드 최적화:** WAFv2 규칙(`field_to_match`) 구문 오류 수정 및 코드 표준화 완료.
*   **배포 완료:** HTTP API Gateway 기반의 서버리스 오리진 배포 성공 (`Apply complete!`).
*   **서비스 상태:** CloudFront 도메인(`dh65zwirzigtb.cloudfront.net`)을 통한 정상 서비스 가동 중.

---

### 2. 주요 트러블슈팅 기록 (Key Troubleshooting)

| 발생 이슈 | 원인 분석 | 해결 조치 |
| :--- | :--- | :--- |
| **DependencyViolation** | Root 모듈 내 교차 의존성(SG Rule, WAF Association)으로 인한 삭제 거부 | `-target` 옵션을 활용한 **4단계 전략적 파괴 프로세스**로 의존성 순차 해제 |
| **Missing Argument (format)** | API GW Stage 로깅 설정 시 필수 인자인 `format` 누락 | `jsonencode` 함수를 활용하여 구조화된 JSON 로깅 포맷 정의 |
| **KMS AccessDenied** | CloudWatch Logs의 KMS 키 암호화 접근 권한 부재 | 우선순위가 낮은 로그 그룹의 KMS 설정을 제거하고 AWS 기본 암호화로 대체 |
| **Athena AccessDenied** | 쿼리 실행(`StartQueryExecution`)에 필요한 IAM 권한 부족 | **(기상 후 조치)** IAM 계정에 Athena 및 S3 결과 저장 권한 부여 필요 |

### 3. 사후 점검 및 기상 후 진행 업무 (Next Step)

#### **① CloudFront-API Gateway 연결 최종 확인**
*   브라우저에서 `dh65zwirzigtb.cloudfront.net` 접속 시 `SecurityAnalyzer` Lambda의 응답이 정상적으로 출력되는지 확인.

#### **② 실시간 로그 적재 및 분석 확인**
*   **로그 경로:** `/aws/apigateway/devsecops-platform-http-api`
*   **체크 방법:** CloudWatch Logs Console에서 신규 로그 스트림 생성 여부 확인.
*   **권한 복구:** 작업 중 발생한 Athena 권한 에러를 해결하기 위해 IAM 정책에 `AmazonAthenaFullAccess` 추가 후 **Logs Insights** 쿼리 테스트 진행.
