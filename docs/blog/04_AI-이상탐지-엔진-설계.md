## WAF 룰이 놓치는 것

WAF Managed Rule은 알려진 공격 패턴을 정적으로 정의합니다. SQL Injection이면 `UNION SELECT`, XSS면 `<script>` 같은 식입니다.

문제는 공격자가 이 패턴을 알고 있다는 겁니다.

`%3cscript%3e`는 `<script>`를 URL 인코딩한 것입니다. WAF CommonRuleSet이 이를 잡긴 하지만, 인코딩 레이어를 두 번 이상 중첩하거나, 정상 요청 사이에 낮은 빈도로 섞인 공격은 룰 기반 시스템이 쉽게 통과시킵니다.

이런 요청들의 공통점은 **구조가 비정상적**이라는 겁니다. URI 길이가 비정상적으로 길거나, 문자 분포가 정상 요청과 다릅니다. 규칙으로 잡기는 어렵지만, 분포 기반 비지도 학습으로 잡을 수 있습니다.

## 왜 Isolation Forest인가

세 가지를 검토했습니다.

| 알고리즘 | 장점 | 단점 |
|:---|:---|:---|
| **Isolation Forest** | 비교적 빠른 학습·추론 속도, 비지도 학습 | 국소적 이상치 탐지에 약함 |
| Autoencoder | 정교한 피처 추출 | 학습 리소스 과다, 튜닝 어려움 |
| One-Class SVM | 비선형 경계를 잘 찾음 | 데이터 커질수록 추론 속도 급락 |

WAF 로그는 실시간으로 쌓입니다. 추론은 요청이 들어올 때마다 발생합니다. Autoencoder와 One-Class SVM은 데이터가 늘어날수록 병목이 생기는 구조입니다.

Isolation Forest는 "정상 데이터 포인트는 분리하기 어렵고, 이상치는 분리하기 쉽다"는 가정 위에서 동작합니다. 트리를 무작위로 분기해 포인트를 고립시키는 데 필요한 분기 수가 적을수록 이상치로 판단합니다. 트리 기반 구조라 비교적 빠른 추론 속도를 유지할 수 있고, 레이블 없이 학습할 수 있어 WAF 로그처럼 공격 여부가 명확하지 않은 데이터에 적합합니다.

## 피처 설계 — path_entropy와 args_entropy, query string을 분리한 이유

처음엔 피처 3개로 시작했습니다: `country_code`, `rule_code`, `uri_len`.

URI 길이만으로는 공격을 구분하기 어렵다는 점에서 Shannon Entropy를 추가하려 했습니다.

$$H(X) = -\sum_{i} p(x_i) \log_2 p(x_i)$$

```python
def calculate_entropy(text: str) -> float:
    if not text:
        return 0.0
    freq = Counter(text)
    n = len(text)
    return -sum((c / n) * math.log2(c / n) for c in freq.values())
```

여기서 한 가지 문제를 발견했습니다. AWS WAF 로그는 URI path와 query string을 분리해서 기록합니다.

```
실제 공격 요청: /?id=1' UNION SELECT username, password FROM users--

WAF 로그 기록:
  httpRequest.uri  → "/"          ← path만 저장
  httpRequest.args → "id=1' UNION SELECT ..."  ← query string은 별도
```

SQL 키워드, 특수문자, 인코딩 문자열이 혼합되면 문자 분포가 복잡해져 Shannon Entropy가 증가합니다. 이 신호가 args에 집중되어 있다는 점이 핵심입니다.

전처리 과정에서 `args`를 누락하면 공격 URI가 모두 `/`로 기록됩니다. 단순히 URI 전체에 엔트로피를 적용하면 `/`의 엔트로피는 0이 되어 정상 트래픽과의 구분력이 크게 떨어집니다.

해결 방향은 **path와 args를 분리해 각각 엔트로피를 측정**하는 것이었습니다.

| 구분 | path_entropy | args_entropy |
|:---|:---:|:---:|
| `/api/users/profile` (정상) | ≈ 3.2 | 0.0 |
| `/` + SQLi 페이로드 (공격) | 0.0 | **≈ 4.6** |
| `/` + XSS 페이로드 (공격) | 0.0 | **≈ 4.5~5.0** |

공격은 path가 아닌 query string에서 발생합니다. `args_entropy`가 높을수록 인코딩된 페이로드가 포함된 가능성이 큽니다. 이 구조적 발견이 피처 설계의 핵심이었습니다.

이를 통해 로그 수집 단계에서 `args`를 보존하는 것이 feature engineering의 전제 조건임을 확인했습니다. 프로덕션 환경에서는 `httpRequest.uri + "?" + httpRequest.args`를 연결한 full URI 기반으로 엔트로피를 계산하는 것이 권장됩니다.

최종 피처는 5개입니다: `country_code`, `rule_code`, `uri_len`, `path_entropy`, `args_entropy`.

## 학습 데이터의 진화 — 공격만 보여준 모델의 한계

초기 학습 데이터는 Nikto로 생성한 공격 로그 400건이었습니다.

문제가 있었습니다. Isolation Forest가 이상치를 찾으려면 "정상"이 무엇인지 알아야 합니다. 공격 데이터만으로 학습하면 모델이 "정상 요청의 기준"을 배우지 못합니다. contamination 설정도 신뢰하기 어렵습니다. "이 데이터 중 몇 퍼센트가 이상치인가?"라는 질문에 400건 전부가 공격 로그라면 답이 없습니다.

해결 방향은 하나였습니다. 정상 트래픽을 섞는 것.

실제 WAF ALLOW 로그가 없었기 때문에 합성 데이터를 생성했습니다.

```python
# 정상 트래픽의 조건:
# - source_country = KR (country_code = 0)
# - terminatingruleid = Default_Action (rule_code = 2, ALLOW)
# - 정상적인 URI 구조 (낮은 엔트로피)

NORMAL_URIS = [
    "/", "/index.html", "/login", "/api/users",
    "/api/health", "/static/css/main.css", ...
]
```

공격 로그 400건에 정상 트래픽 1,400건을 추가해 1,800건 혼합 데이터셋을 구성했습니다. 비율은 공격 약 22%, 정상 약 78%입니다.

합성 데이터의 한계는 명확합니다. 실제 사용자 행동 패턴과 다를 수 있습니다. 이 부분은 아래 한계 섹션에서 다시 이야기합니다.

## contamination 조정 근거 — 0.15에서 0.25로

`contamination`은 Isolation Forest에 "이 데이터에서 이상치가 차지하는 비율이 얼마냐"를 알려주는 하이퍼파라미터입니다.

초기 설정은 `0.15`였습니다. 공격 로그 400건 중 약 14.8%를 이상치로 보는 설정이었습니다.

데이터를 1,800건으로 확장한 후 계산이 달라졌습니다.

```
공격 로그 400건 / 전체 1,800건 ≈ 22.2%
→ contamination = 0.25 (오차 여유 포함)
```

단순히 비율만 맞춘 건 아닙니다. contamination별 SQLi / 정상 트래픽 점수 분리도를 비교해 선정 기준을 검증했습니다.

```
contamination=0.25 기준:
  이상 탐지: 427건 / 1,800건 (23.7%)
  SQLi avg score : -0.098  (음수 = 이상 구간)
  지역차단 avg   :  0.022  (양수 = 정상 구간)
  Score IQR      :  0.0491
```

flagged 비율이 20% 미만인 구간은 실제 운영 환경에서 공격 후보를 지나치게 적게 탐지하는 경향이 있었습니다. SQLi avg가 안정적으로 음수 구간을 유지하면서, flagged 비율이 처음으로 20~30% 범위에 진입한 0.25를 권장값으로 선택했습니다.

IQR 0.0491은 0.05 기준선을 소폭 하회합니다. 합성 데이터 특성상 점수 분포가 좁게 측정되는 구조적 한계로 해석했습니다. 실제 WAF 로그 적용 시 IQR 개선이 예상됩니다.

## Lambda 연동 — 탐지에서 차단까지

모델 추론 결과가 WAF IP Set에 자동으로 반영되는 전체 흐름입니다.

```
monitor.py (로컬/EC2)
  → Isolation Forest로 이상 탐지
  → S3 results/*.json 업로드

S3 Event Notification
  → Analyzer Lambda (SecurityAnalyzer) 트리거

Analyzer Lambda
  → Athena로 anomaly=1 IP 집계
  → Preventer Lambda 호출 (ip 키로 전달)

Preventer Lambda (SecurityPreventer)
  → WAF UpdateIPSet API 호출
  → devsecops-ai-block-list에 IP 추가
  → CloudWatch 지표 기록
```

Analyzer Lambda가 Preventer를 호출할 때 페이로드 키가 일치해야 합니다.

```python
# Analyzer Lambda
payload = {
    "ip": target_ip,       # preventer가 기대하는 키
    "count": attack_count,
    "message": "AI-based Anomaly Detected"
}

# Preventer Lambda
attack_ip = event.get("ip")   # 동일한 키
```

Preventer Lambda는 Terraform으로 별도 zip으로 패키징합니다. Analyzer와 소스 디렉토리를 분리하지 않으면 `lambda_security_preventer.handler`를 찾지 못합니다.

```hcl
data "archive_file" "preventer_zip" {
  type        = "zip"
  source_dir  = "${path.root}/../lambda/preventer"
  output_path = "${path.root}/../lambda/preventer.zip"
}

resource "aws_lambda_function" "preventer" {
  filename         = data.archive_file.preventer_zip.output_path
  source_code_hash = data.archive_file.preventer_zip.output_base64sha256
  handler          = "lambda_security_preventer.handler"
}
```

## 재학습 결과 검증

재학습 모델이 실제로 정상과 공격을 구분하는지 확인했습니다.

```python
model  = joblib.load("ai/models/isolation_forest_model.pkl")
scaler = joblib.load("ai/models/scaler.pkl")

# 정상 요청 (KR, ALLOW, 짧은 URI path, args 없음)
normal = {"country_code": 0, "rule_code": 2, "uri_len": 12,
          "path_entropy": 3.0, "args_entropy": 0.0}
# score=양수 → NORMAL

# SQLi 공격 (KR, SQLi 룰 차단, args에 페이로드 포함)
attack = {"country_code": 0, "rule_code": 0, "uri_len": 10,
          "path_entropy": 0.0, "args_entropy": 4.66}
# score=음수 → ANOMALY
```

score가 양수면 정상 구간, 음수면 이상 구간입니다.

성능 평가는 **SQLi 탐지 기준**으로 산정했습니다. GeoBlock(Allow-Only-Korea 룰 기반 차단)은 AI가 새로 탐지해야 하는 공격이 아니라 WAF 룰이 이미 처리한 정책 기반 차단 이벤트이므로 평가에서 제외했습니다.

```
평가 대상: SQLi 100건 + 정상(ALLOW) 1,350건 = 1,500건 (GeoBlock 300건 제외)

Recall    : 1.0000 (100.0%)  — SQLi 100건 전부 탐지, FN=0
Precision : 0.3058  (30.6%)
F1-Score  : 0.4684
FPR       : 0.1621  (16.2%)
```

Recall 100%는 비지도 이상 탐지 특성상 보수적으로 잡는 방향으로 동작한 결과입니다. WAF 정적 룰 기반 탐지를 보완해 공격 미탐(FN)을 줄이는 전략이며, FP 증가와 Recall 향상 사이의 trade-off입니다.

## 한계 인정

이 모델이 모든 공격을 잡는다고 말하면 거짓말입니다.

**FP 증가와 Recall 향상의 trade-off**: 현재 모델은 SQLi Recall 100%, FPR 16.2%로 동작합니다. 공격 미탐(FN=0)을 우선하는 보수적 탐지 전략의 결과입니다. WAF 정적 룰 기반 탐지를 보완하는 2차 레이어이므로, FP로 인한 추가 검토 비용과 미탐 방지 효과 사이의 운영 판단이 필요합니다.

**오탐 가능성**: Base64 파라미터나 복잡한 검색 쿼리를 사용하는 정상 API 요청은 args_entropy가 높게 나올 수 있습니다. 모델은 이를 이상치로 볼 수 있습니다.

**저속 공격(Low-and-slow) 탐지 어려움**: Isolation Forest는 개별 요청의 피처 분포를 봅니다. 낮은 빈도로 오랜 시간에 걸쳐 진행되는 스캔은 개별 요청 하나만 보면 정상처럼 보일 수 있습니다.

**WAF 로그 정규화 한계**: AWS WAF 로그는 URI path와 query string을 분리 저장합니다. 전처리 단계에서 args를 보존하지 않으면 공격 페이로드의 복잡도가 소실됩니다. 현재 실험에서는 rule type 기반 대표 payload 패턴을 사용해 args_entropy 분포를 재현했습니다. 실제 운영 환경에서는 `httpRequest.args`를 직접 파싱해 사용해야 합니다.

**합성 데이터 한계**: 정상 트래픽 1,400건은 실제 사용자 행동이 아닌 생성된 데이터입니다. 실제 WAF ALLOW 로그를 수집하면 모델의 정상 기준이 더 정확해질 것입니다.

그래서 이 엔진은 단독으로 동작하지 않습니다.

```
WAF Managed Rules (정적 분석)
  → 알려진 패턴의 공격을 1차 차단

AI Isolation Forest (분포 기반 탐지)
  → 패턴 우회, 변조된 공격을 2차 탐지

두 레이어가 서로의 약점을 보완하는 구조
```

WAF는 알려진 패턴 기반 차단을 담당하고, AI는 분포 기반으로 추가 이상 징후를 탐지한다. 두 레이어는 서로 다른 기준으로 동작한다.

## 마치며

이 엔진에서 내린 판단들을 다시 정리하면 이렇습니다.

Isolation Forest를 선택한 건 가장 정교한 알고리즘이어서가 아닙니다. WAF 로그라는 실시간 비지도 환경에서 추론 속도와 레이블 불필요라는 조건에 가장 맞는 선택이었습니다.

Shannon Entropy를 `path_entropy`와 `args_entropy`로 분리한 건 `uri_len`만으로 URI 인코딩 공격을 구분할 수 없다는 문제에서 시작했습니다. AWS WAF 로그가 path와 query string을 분리 저장한다는 구조를 파악한 뒤, 공격 페이로드의 복잡도가 args에 집중된다는 점을 피처 설계에 반영했습니다. 엔트로피는 단일 문자열의 내재적 복잡도를 측정하므로 샘플 수에 덜 의존적이라는 점도 소량 데이터 환경에서 유리했습니다.

contamination을 0.25로 설정한 건 "데이터에서 공격이 차지하는 실제 비율"을 기준으로 삼았습니다. 그리고 IQR로 과적합 여부를 확인했습니다.

합성 정상 트래픽을 섞은 건 모델이 "정상"을 모른 채 이상을 판단하는 구조적 문제를 해소하기 위해서입니다.

그리고 한계를 적었습니다. 이 모델이 잡지 못하는 것, 오탐할 수 있는 상황을 아는 것이 이 엔진을 제대로 운영하는 전제입니다.

결국 이 프로젝트의 목적은 "AI가 WAF를 대체하는 것"이 아니라, 정적 룰 기반 보안을 보완하는 운영형 탐지 레이어를 만드는 것이었습니다.
