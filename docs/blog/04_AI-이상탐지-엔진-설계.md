## WAF 룰이 놓치는 것

WAF Managed Rule은 알려진 공격 패턴을 정적으로 정의합니다. SQL Injection이면 `UNION SELECT`, XSS면 `<script>` 같은 식입니다.

문제는 공격자가 이 패턴을 알고 있다는 겁니다.

`%3cscript%3e`는 `<script>`를 URL 인코딩한 것입니다. WAF CommonRuleSet이 이를 잡긴 하지만, 인코딩 레이어를 두 번 이상 중첩하거나, 정상 요청 사이에 낮은 빈도로 섞인 공격은 룰 기반 시스템이 쉽게 통과시킵니다.

이런 요청들의 공통점은 **구조가 비정상적**이라는 겁니다. URI 길이가 비정상적으로 길거나, 문자 분포가 정상 요청과 다릅니다. 규칙으로 잡기는 어렵지만, 분포 기반 비지도 학습으로 잡을 수 있습니다.

## 왜 Isolation Forest인가

세 가지를 검토했습니다.

| 알고리즘 | 장점 | 단점 |
|:---|:---|:---|
| **Isolation Forest** | 선형 복잡도, 비지도 학습, 빠른 추론 | 국소적 이상치 탐지에 약함 |
| Autoencoder | 정교한 피처 추출 | 학습 리소스 과다, 튜닝 어려움 |
| One-Class SVM | 비선형 경계를 잘 찾음 | 데이터 커질수록 추론 속도 급락 |

WAF 로그는 실시간으로 쌓입니다. 추론은 요청이 들어올 때마다 발생합니다. Autoencoder와 One-Class SVM은 데이터가 늘어날수록 병목이 생기는 구조입니다.

Isolation Forest는 "정상 데이터 포인트는 분리하기 어렵고, 이상치는 분리하기 쉽다"는 가정 위에서 동작합니다. 트리를 무작위로 분기해 포인트를 고립시키는 데 필요한 분기 수가 적을수록 이상치로 판단합니다. 추론이 O(n log n)으로 빠르고, 레이블 없이 학습할 수 있어 WAF 로그처럼 공격 여부가 명확하지 않은 데이터에 적합합니다.

## 피처 설계 — Shannon Entropy를 추가한 이유

처음엔 피처 3개로 시작했습니다: `country_code`, `rule_code`, `uri_len`.

문제는 URI 길이만으로는 구분이 어렵다는 점이었습니다.

```
/api/users/profile           → uri_len = 18
/?id=1%27%20UNION%20SELECT   → uri_len = 27
```

길이 차이는 있지만, 공격자가 padding을 추가하면 길이로는 구분이 안 됩니다. 반면 **문자 분포**는 다릅니다.

Shannon Entropy는 문자열 내 문자 분포의 불확실성을 측정합니다.

$$H(X) = -\sum_{i} p(x_i) \log_2 p(x_i)$$

```python
def calculate_entropy(text: str) -> float:
    if not text:
        return 0.0
    freq = Counter(text)
    n = len(text)
    return -sum((c / n) * math.log2(c / n) for c in freq.values())
```

실제로 측정하면 이렇습니다:

| URI | 엔트로피 |
|:---|:---:|
| `/api/users/profile` | ≈ 3.2 |
| `/?id=1' UNION SELECT--` | ≈ 3.9 |
| URL 인코딩된 XSS 페이로드 | ≈ 4.5~5.5 |

정상 URI는 반복되는 구조가 있어 엔트로피가 낮습니다. 공격 페이로드는 다양한 특수문자와 인코딩이 섞여 엔트로피가 높습니다. 이 차이가 피처로서의 가치입니다.

`uri_entropy`를 추가해 최종 피처는 4개가 됐습니다: `country_code`, `rule_code`, `uri_len`, `uri_entropy`.

## 학습 데이터의 진화 — 공격만 보여준 모델의 한계

초기 학습 데이터는 Nikto로 생성한 공격 로그 400건이었습니다.

문제가 있었습니다. Isolation Forest가 이상치를 찾으려면 "정상"이 무엇인지 알아야 합니다. 공격 데이터만으로 학습하면 모델이 "정상 요청의 기준"을 배우지 못합니다. contamination 설정도 신뢰하기 어렵습니다. "이 데이터 중 몇 퍼센트가 이상치인가?"라는 질문에 400건 전부가 공격 로그라면 답이 없습니다.

해결 방향은 하나였습니다. 정상 트래픽을 섞는 것.

실제 WAF ALLOW 로그가 없었기 때문에 합성 데이터를 생성했습니다.

```python
# 정상 트래픽의 조건:
# - source_country = KR (country_code = 0)
# - terminatingruleid = Default_Action (rule_code = 2, ALLOW)
# - 정상적인 URI 구조 (낮은 엔트로피)

NORMAL_URIS = [
    "/", "/index.html", "/login", "/api/users",
    "/api/health", "/static/css/main.css", ...
]
```

공격 로그 400건에 정상 트래픽 1,400건을 추가해 1,800건 혼합 데이터셋을 구성했습니다. 비율은 공격 약 22%, 정상 약 78%입니다.

합성 데이터의 한계는 명확합니다. 실제 사용자 행동 패턴과 다를 수 있습니다. 이 부분은 아래 한계 섹션에서 다시 이야기합니다.

## contamination 조정 근거 — 0.15에서 0.25로

`contamination`은 Isolation Forest에 "이 데이터에서 이상치가 차지하는 비율이 얼마냐"를 알려주는 하이퍼파라미터입니다.

초기 설정은 `0.15`였습니다. 공격 로그 400건 중 약 14.8%를 이상치로 보는 설정이었습니다.

데이터를 1,800건으로 확장한 후 계산이 달라졌습니다.

```
공격 로그 400건 / 전체 1,800건 ≈ 22.2%
→ contamination = 0.25 (오차 여유 포함)
```

단순히 비율만 맞춘 건 아닙니다. 과적합 여부를 IQR로 진단했습니다.

```
이상 탐지: 345건 / 1,800건 (19.2%)
Score IQR: 0.0611
```

IQR이 0.05 미만이면 점수가 특정 구간에 몰려 있다는 신호, 즉 과적합 징후입니다. 0.0611은 이 기준을 넘어 모델이 정상과 이상을 충분히 구분하고 있다는 의미입니다.

## Lambda 연동 — 탐지에서 차단까지

모델 추론 결과가 WAF IP Set에 자동으로 반영되는 전체 흐름입니다.

```
monitor.py (로컬/EC2)
  → Isolation Forest로 이상 탐지
  → S3 results/*.json 업로드

S3 Event Notification
  → Analyzer Lambda (SecurityAnalyzer) 트리거

Analyzer Lambda
  → Athena로 anomaly=1 IP 집계
  → Preventer Lambda 호출 (ip 키로 전달)

Preventer Lambda (SecurityPreventer)
  → WAF UpdateIPSet API 호출
  → devsecops-ai-block-list에 IP 추가
  → CloudWatch 지표 기록
```

Analyzer Lambda가 Preventer를 호출할 때 페이로드 키가 일치해야 합니다.

```python
# Analyzer Lambda
payload = {
    "ip": target_ip,       # preventer가 기대하는 키
    "count": attack_count,
    "message": "AI-based Anomaly Detected"
}

# Preventer Lambda
attack_ip = event.get("ip")   # 동일한 키
```

Preventer Lambda는 Terraform으로 별도 zip으로 패키징합니다. Analyzer와 소스 디렉토리를 분리하지 않으면 `lambda_security_preventer.handler`를 찾지 못합니다.

```hcl
data "archive_file" "preventer_zip" {
  type        = "zip"
  source_dir  = "${path.root}/../lambda/preventer"
  output_path = "${path.root}/../lambda/preventer.zip"
}

resource "aws_lambda_function" "preventer" {
  filename         = data.archive_file.preventer_zip.output_path
  source_code_hash = data.archive_file.preventer_zip.output_base64sha256
  handler          = "lambda_security_preventer.handler"
}
```

## 재학습 결과 검증

재학습 모델이 실제로 정상과 공격을 구분하는지 확인했습니다.

```python
model  = joblib.load("ai-security/models/isolation_forest_model.pkl")
scaler = joblib.load("ai-security/models/scaler.pkl")

# 정상 요청 (KR, ALLOW, 짧은 URI, 낮은 엔트로피)
normal = {"country_code": 0, "rule_code": 2, "uri_len": 12, "uri_entropy": 1.58}
# score=0.0598 → NORMAL

# 공격 시뮬레이션 (비KR, 알 수 없는 룰, 긴 URI, 높은 엔트로피)
attack = {"country_code": 1, "rule_code": 9001, "uri_len": 3500, "uri_entropy": 4.9}
# score=-0.1117 → ANOMALY
```

score가 양수면 정상 구간, 음수면 이상 구간입니다. 재학습 전(contamination=0.2, 400건 공격-only)과 비교해 정상 트래픽이 ANOMALY로 잘못 판단되는 빈도가 줄었습니다.

## 한계 인정

이 모델이 모든 공격을 잡는다고 말하면 거짓말입니다.

**오탐 가능성**: 검색 쿼리가 많거나 Base64 파라미터를 사용하는 정상 API 요청은 엔트로피가 높게 나올 수 있습니다. 모델은 이를 이상치로 볼 수 있습니다.

**저속 공격(Low-and-slow) 탐지 어려움**: Isolation Forest는 개별 요청의 피처 분포를 봅니다. 낮은 빈도로 오랜 시간에 걸쳐 진행되는 스캔은 개별 요청 하나만 보면 정상처럼 보일 수 있습니다.

**합성 데이터 한계**: 정상 트래픽 1,400건은 실제 사용자 행동이 아닌 생성된 데이터입니다. 실제 WAF ALLOW 로그를 수집하면 모델의 정상 기준이 더 정확해질 것입니다.

그래서 이 엔진은 단독으로 동작하지 않습니다.

```
WAF Managed Rules (정적 분석)
  → 알려진 패턴의 공격을 1차 차단

AI Isolation Forest (분포 기반 탐지)
  → 패턴 우회, 변조된 공격을 2차 탐지

두 레이어가 서로의 약점을 보완하는 구조
```

WAF 룰이 시그니처 기반이라면, AI는 분포 기반입니다. 룰이 놓친 것을 AI가 잡고, AI가 오탐한 것은 WAF 룰이 이미 판단을 끝낸 상태입니다.

## 마치며

이 엔진에서 내린 판단들을 다시 정리하면 이렇습니다.

Isolation Forest를 선택한 건 가장 정교한 알고리즘이어서가 아닙니다. WAF 로그라는 실시간 비지도 환경에서 추론 속도와 레이블 불필요라는 조건에 가장 맞는 선택이었습니다.

Shannon Entropy를 추가한 건 `uri_len`만으로 URI 인코딩 공격을 구분할 수 없다는 문제에서 시작했습니다. 엔트로피는 단일 문자열의 내재적 복잡도를 측정하므로 샘플 수에 덜 의존적이라는 점도 소량 데이터 환경에서 유리했습니다.

contamination을 0.25로 설정한 건 "데이터에서 공격이 차지하는 실제 비율"을 기준으로 삼았습니다. 그리고 IQR로 과적합 여부를 확인했습니다.

합성 정상 트래픽을 섞은 건 모델이 "정상"을 모른 채 이상을 판단하는 구조적 문제를 해소하기 위해서입니다.

그리고 한계를 적었습니다. 이 모델이 잡지 못하는 것, 오탐할 수 있는 상황을 아는 것이 이 엔진을 제대로 운영하는 전제입니다.
