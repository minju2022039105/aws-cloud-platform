## 1. 배포의 역설

1편에서 WAF 5단계 규칙과 AI 이상 탐지 구조를 설계했다. Geo-Blocking, SQLi 방어, AI 기반 실시간 IP 차단까지. 꽤 그럴싸한 보안 아키텍처다.

그런데 문득 이런 생각이 들었다.

> "보안 인프라를 열심히 코드로 짜놨는데, 정작 그 코드를 AWS에 배포할 때는 어떻게 하고 있지?"

만약 CI/CD 파이프라인에서 IAM 액세스 키를 GitHub Secrets에 저장해두고 쓰고 있다면? 그 키가 유출되는 순간 WAF고 AI고 다 무의미해진다. 공격자가 직접 인프라를 수정할 수 있으니까.

또 다른 문제도 있다. Terraform 코드 한 줄 잘못 건드리면 WAF 규칙이 통째로 날아갈 수 있다. 배포 전에 그 코드가 안전한지 누가 검사하는가?

**보안 아키텍처 자체를 지키는 보안**이 필요했다. 그게 이번 글의 주제다.

---

## 2. 파이프라인 전체 구조

파이프라인은 3개의 Job으로 구성된다.

```
Security Gates → Terraform → Lambda 배포
```

GitHub Actions의 `needs:` 키워드로 순서를 강제했다.

```yaml
jobs:
  security-scan:
    name: Security Gates
    ...

  terraform:
    name: Terraform
    needs: security-scan   # security-scan이 성공해야만 실행
    ...

  deploy-lambda:
    name: Deploy Lambda
    needs: terraform       # terraform이 성공해야만 실행
    ...
```

이 구조의 핵심은 **앞 단계가 실패하면 뒤가 실행되지 않는다**는 것이다. 보안 게이트에서 HIGH 취약점이 발견되면 Terraform Apply 자체가 막힌다. 코드가 AWS에 닿기 전에 파이프라인이 멈춘다.

## 3. 검문소 세우기 — Trivy + Bandit

### Trivy: 인프라 코드 스캔

```yaml
- name: Trivy IaC Scan (HIGH/CRITICAL 차단)
  uses: aquasecurity/trivy-action@master
  with:
    scan-type: config
    scan-ref: ./infra
    severity: HIGH,CRITICAL
    exit-code: 1
```

`./infra` 디렉토리의 Terraform 코드를 스캔해서 HIGH/CRITICAL 등급 취약점이 발견되면 `exit-code: 1`로 파이프라인을 중단한다.

처음 이 설정을 켰을 때, 파이프라인이 바로 죽었다. 예상은 했지만 4개의 항목이 한꺼번에 걸렸다.

### 오탐 관리: .trivyignore

여기서 중요한 판단이 필요했다. Trivy가 잡아낸 4개의 항목이 진짜 취약점인가, 아니면 설계상 의도된 동작인가?

무조건 고치는 게 답이 아니다. 예를 들어 ALB를 퍼블릭으로 노출하는 건 외부 트래픽을 받아야 하는 서버 입장에서 당연한 설정이다. 문제는 그걸 WAF 없이 노출하는 것이지, 노출 자체가 아니다.

그래서 `.trivyignore` 파일을 만들고, 각 항목마다 **"왜 무시하는지"를 문서화**했다.

```bash
# [AVD-AWS-0053] ALB가 인터넷에 노출됨 (internal = false)
# 설계 의도: 이 ALB는 외부 사용자 트래픽을 수신하는 퍼블릭 엔드포인트.
#            WAF Web ACL이 앞단에서 Geo-Blocking + AI 기반 IP 차단을 수행.
AVD-AWS-0053

# [AVD-AWS-0104] SG egress 규칙이 0.0.0.0/0 허용
# 설계 의도: EC2가 AWS API(CloudWatch, S3), apt 패키지 등에 접근해야 함.
#            VPC Flow Logs + CloudTrail로 아웃바운드 트래픽 전량 감사 중.
AVD-AWS-0104
```

이렇게 하지 않으면 나중에 문제가 생긴다. 6개월 뒤에 `.trivyignore`를 열어봤을 때 "이게 왜 무시되어 있지?"를 알 수 없게 된다. 진짜 취약점이 슬쩍 끼어들어도 구분이 안 된다.

**무시(ignore)와 문서화(document)는 다르다.** ignore만 하는 건 눈 감는 것이고, 문서화를 함께 하는 건 의식적인 판단이다.

### Bandit: Python 코드 정적 분석

```yaml
- name: Bandit Python Scan
  run: |
    pip install bandit -q
    bandit -r ./ai ./lambda -ll --exit-zero
```

Lambda 함수와 AI 모델 코드를 정적 분석한다. `-ll`은 MEDIUM 이상만 보고하는 옵션이고, `--exit-zero`는 발견해도 파이프라인을 멈추지 않는다.

Trivy와 다르게 `--exit-zero`를 쓴 이유는 Bandit이 Python 코드 특성상 오탐률이 높기 때문이다. 발견 내역은 로그에 남기되, 배포를 막지는 않는다. 향후 오탐 패턴이 파악되면 `.banditignore`로 관리할 계획이다.

## 4. OIDC 인증과 tfstate 보안

### 왜 액세스 키를 쓰지 않는가

일반적인 CI/CD 설정에서는 AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY를 GitHub Secrets에 넣는다. 단순하고 빠르다. 하지만 두 가지 문제가 있다.

첫째, 장기 자격증명(Long-lived credentials)이다. 키는 만료되지 않는다. 유출되면 삭제하기 전까지 계속 유효하다.

둘째, 키 자체가 공격 대상이 된다. GitHub Secrets가 뚫리거나, 실수로 로그에 출력되거나, 워크플로우 파일에 하드코딩되는 사고가 실제로 발생한다.

OIDC(OpenID Connect)는 이 문제를 구조적으로 없앤다.

```yaml
- name: Configure AWS credentials (OIDC)
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
    aws-region: us-east-1
```

동작 원리는 이렇다. GitHub Actions가 실행될 때 GitHub의 OIDC Provider가 JWT 토큰을 발급한다. AWS는 이 토큰을 신뢰하도록 사전에 설정된 IAM Role로 일시적인 자격증명을 내준다. 이 자격증명은 해당 Job이 끝나면 만료된다.

키를 저장할 필요가 없고, 유출되더라도 곧 만료된다.

### tfstate 파일 보안

Terraform을 쓰면 반드시 마주치는 문제가 있다. `terraform.tfstate` 파일에는 인프라의 현재 상태가 담기는데, 여기에 민감한 정보가 평문으로 포함될 수 있다.

이 프로젝트의 경우 KMS 키 ARN, S3 버킷 이름, 보안 그룹 ID 등이 tfstate에 들어간다. 이 파일이 로컬에 남거나 Git에 커밋되면 안 된다.

S3 Remote Backend + DynamoDB Lock으로 해결했다.

```hcl
terraform {
  backend "s3" {
    bucket         = "minju-devsecops-tfstate-virginia"
    key            = "devsecops/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-lock-table"
    encrypt        = true
  }
}
```

- **S3 저장**: tfstate가 로컬에 남지 않는다
- **encrypt = true**: S3에 저장할 때 서버사이드 암호화
- **DynamoDB Lock**: 여러 사람이 동시에 `terraform apply`를 실행해도 state 파일이 충돌하지 않는다

### PR에서 인프라 변경 미리 보기

```yaml
- name: PR에 Plan 결과 코멘트
  if: github.event_name == 'pull_request'
  uses: actions/github-script@v7
  with:
    script: |
      const plan = fs.readFileSync(process.env.GITHUB_WORKSPACE + '/plan_output.txt', 'utf8');
      github.rest.issues.createComment({
        issue_number: context.issue.number,
        body: `## Terraform Plan\n\`\`\`\n${plan}\n\`\`\``
      });
```

PR을 열면 Terraform Plan 결과가 자동으로 코멘트로 달린다. 리뷰어가 "이 PR이 인프라를 어떻게 바꾸는지"를 코드 리뷰와 함께 확인할 수 있다.

Apply는 main 브랜치에 push될 때만 실행된다.

```yaml
- name: Terraform Apply
  if: github.event_name == 'push' && github.ref == 'refs/heads/main'
  run: terraform apply -auto-approve
```

PR 단계에서는 Plan만, 머지 후에 Apply. 인프라 변경도 코드 리뷰 프로세스 안에서 관리된다.

## 5. Lambda 배포 방식의 선택과 트레이드오프

### Terraform이 이미 Lambda를 배포한다

`lambda.tf`를 보면 Terraform에 이미 `source_code_hash`가 설정되어 있다.

```hcl
resource "aws_lambda_function" "analyzer" {
  filename         = data.archive_file.lambda_zip.output_path
  source_code_hash = data.archive_file.lambda_zip.output_base64sha256
  ...
}
```

`source_code_hash`는 zip 파일의 해시값이다. 코드가 바뀌면 해시가 달라지고, Terraform이 이를 감지해서 Lambda를 재배포한다. Terraform Apply만 해도 Lambda 코드가 반영된다는 뜻이다.

그럼에도 Job 3에서 `update-function-code`를 별도로 실행한다.

```yaml
- name: Deploy SecurityAnalyzer
  run: |
    cd lambda/analyzer
    zip -r analyzer.zip .
    aws lambda update-function-code \
      --function-name SecurityAnalyzer \
      --zip-file fileb://analyzer.zip
```

### 왜 별도 Job으로 분리했는가

두 가지 이유다.

첫째, **인프라 변경 없이 Lambda 코드만 수정할 때**다. WAF 규칙이나 API Gateway 설정은 그대로인데 Lambda 로직만 바꾸는 경우, Terraform은 인프라 리소스에 변경이 없으면 `source_code_hash`를 체크하지 않고 no-change로 넘어갈 수 있다. 별도 Job은 이런 엣지 케이스를 커버한다.

둘째, **배포 가시성**이다. Terraform Apply 로그에 Lambda 배포가 묻히지 않고, Job 3 로그에서 "어떤 Lambda가 언제 배포됐는지"를 명확하게 확인할 수 있다.

### 현재 방식의 한계

솔직히 말하면 이 구조는 개선 여지가 있다.

`update-function-code`는 새 코드를 바로 $LATEST에 반영한다. 배포 도중 에러가 나도 자동 롤백이 없다. 트래픽이 많은 서비스라면 Lambda Alias와 가중치 기반 트래픽 전환(Canary 배포)을 쓰는 게 맞다.

지금은 포트폴리오 규모의 트래픽이라 단순하게 유지했다. 프로덕션 수준으로 가져간다면 이 부분이 첫 번째 개선 대상이다.

## 6. 마치며

이 파이프라인을 만들면서 가장 크게 배운 건 **"보안은 코드가 실행된 후가 아니라 커밋 시점에 시작된다"** 는 것이다.

Trivy가 Terraform 코드를 스캔하고, OIDC가 키 유출 가능성을 없애고, tfstate가 암호화된 S3에 보관되는 것 — 이 모든 게 코드가 AWS에 닿기 전에 이뤄진다.

`.trivyignore`에 이유를 문서화하는 작업도 마찬가지다. "이건 취약점이 아니라 설계다"라는 판단을 기록으로 남기는 것. 나중에 팀원이 보더라도, 6개월 뒤의 내가 보더라도 맥락이 남아있어야 한다.
