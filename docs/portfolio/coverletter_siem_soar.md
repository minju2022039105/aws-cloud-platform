## 자기소개 및 지원동기

저는 로그를 수집하는 것을 넘어, 그 결과가 실제 차단 행동으로 이어지는 구조를 직접 설계해온 엔지니어입니다.

부트캠프 팀 프로젝트에서 AWS WAF 로그를 Isolation Forest로 분석해 공격을 60초 먼저 탐지하고, GuardDuty → EventBridge → Lambda → WAF IPSet으로 이어지는 자동 차단 파이프라인을 구현했습니다. 개인 프로젝트에서는 Kinesis Firehose → S3 → Athena로 WAF 원본 로그를 수집·분석하고, S3 이벤트 트리거 기반 SOAR 파이프라인으로 이상 탐지 결과를 차단 정책에 자동 반영했습니다.

그 과정에서 "로그를 어디에 저장할 것인가"보다 "이 로그가 어떻게 탐지와 대응으로 연결되는가"를 먼저 고민하게 됐고, 그것이 SIEM/SOAR 직무로 이어진 이유입니다.

메가존클라우드가 Splunk, Panther, Google SecOps 등 글로벌 SIEM 플랫폼을 다루고, AI Driven SOC 고도화를 지향한다는 점이 매력적으로 다가왔습니다. 직접 설계한 AI 탐지 파이프라인 경험을 엔터프라이즈 환경의 SIEM/SOAR 플랫폼으로 확장하고 싶다는 확신이 지원을 결심하게 했습니다.

## Trinity Values

### TRANSFORM

#### Customer-First Innovation

SIEM/SOAR 시스템에서 오탐(False Positive)은 단순한 기술 지표가 아니라, 보안 담당자의 알림 피로도로 이어지는 운영 문제입니다.

초기 설계에서는 `pred_risk > 80`이라는 고정 임계값을 사용했습니다. 그런데 트래픽 패턴이 변하면 고정 임계값은 오탐을 급격히 늘린다는 한계를 발견했습니다. 이벤트 기간 트래픽이 폭증하는 이커머스 환경에서 특히 치명적인 문제였습니다.

이를 해결하기 위해 전체 위험 점수 분포의 하위 5%를 동적 임계값으로 설정했습니다. 실시간 트래픽 흐름에 따라 기준이 자동 조정되어, 오탐을 줄이면서도 실제 위협 탐지율은 유지할 수 있었습니다.

```python
threshold = np.percentile(scores_all, 5)
```

고객 환경의 트래픽 패턴은 서비스마다, 시간대마다 다릅니다. 고정된 룰이 아닌 환경에 적응하는 탐지 기준이 실제 운영에서 더 가치 있다는 것을 이 경험에서 배웠습니다.

#### Excellence in Execution

개인 DevSecOps 프로젝트에서 monitor.py의 무한루프가 614만 건의 KMS 호출을 발생시켜 하루 $30.60이 청구됐습니다. 평소 월 $2~3 수준의 계정에서 발생한 일이었습니다.

저는 이 문제를 단순히 수습하는 데서 끝내지 않았습니다. Cost Explorer로 비용 급증을 확인하고, CloudTrail로 API 호출 출처를 역추적해 구조적 원인을 특정했습니다. 이후 KMS 호출 구조를 재설계해 비용을 $18.42에서 $1.41로 약 92% 절감했습니다.

SIEM/SOAR 환경에서는 로그 수집량과 쿼리 빈도가 직접 비용으로 이어집니다. 문제를 구조 단위로 추적하고 재설계하는 이 경험이, 고객 환경에서 로그 파이프라인 비용을 최적화하는 데 실질적으로 적용될 수 있다고 생각합니다.
