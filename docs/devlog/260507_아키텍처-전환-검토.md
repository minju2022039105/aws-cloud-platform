# [260507] 아키텍처 전환 검토 및 프로젝트 정비

## 1. 배경

멘토님 가이드라인(WAF + ALB + EC2)을 실습 비용 절감 및 서버리스 경험 확보 목적으로
WAF + API Gateway + Lambda 구조로 변경하여 배포 시도.
오늘 현재 상태 점검 및 방향 재확인.

---

## 2. 인프라 상태 점검

| 항목 | 상태 |
|------|------|
| AWS 실제 리소스 (ALB, WAF) | 존재함 (CLI 삭제 시도했으나 실제 삭제 안 됨, 콘솔 확인) |
| Terraform state | ALB, WAF, EC2 모두 기록되어 있음 |
| Terraform 코드 | ALB 모듈 + EC2 + apigateway.tf 혼재 |
| WAF 연결 대상 | ALB (API Gateway에는 아직 미연결) |

**결론**: state drift 없음. 코드와 AWS 리소스 일치. 단, 두 아키텍처가 코드에 혼재 중.

### 교훈
AWS 리소스를 CLI로 직접 삭제하면 Terraform state와 불일치(drift) 발생.
앞으로는 `terraform destroy -target` 또는 코드 수정 → `terraform apply` 순서로 관리.

---

## 3. 아키텍처 전환 트레이드오프

### 얻는 것 (API Gateway + Lambda)
- EC2 t3.micro 월 $8-9 고정 비용 → Lambda는 포트폴리오 트래픽 기준 거의 $0
- OS 패치, AMI 관리 등 운영 오버헤드 제거
- WAF 로그 → S3 → Lambda 트리거 흐름이 이벤트 기반 워크로드에 자연스러움

### 포기하는 것 (ALB + EC2)
- ALB L7 헬스체크, 고급 라우팅 기능
- 전통적인 서버 보안 하드닝 실습 경험
- 구조 단순성 (WAF → ALB → EC2 직선 흐름)

**결정**: 인프라 구조 확정은 AI 작업 이후에 한 번에 정리.

## 4. WAF 로그 현황 확인 및 정상 트래픽 수집 파이프라인 구축

### WAF 로그 현황 (S3: aws-waf-logs-minju-0417-project)

| 항목 | 수치 |
|------|------|
| 전체 레코드 | 184건 |
| ALLOW | **0건** |
| BLOCK (GeoBlock-Non-KR) | 180건 |
| BLOCK (IP Reputation) | 4건 |

정상 트래픽 로그가 전혀 없었음. GeoBlock 규칙이 한국 외 IP를 전부 막아서
한국 IP의 ALLOW 요청만 로그에 기록됨 → 실제 서비스 접근이 없으니 ALLOW 로그 0건.

### 해결: NormalTrafficGenerator Lambda 배포

ML 학습용 정상 트래픽 자동 수집을 위해 Lambda + EventBridge 파이프라인 구축.

- **스케줄**: 6시간마다 자동 실행 (하루 4회)
- **1회 생성량**: 시간대별 가중치 적용 후 최대 300건
  - 업무시간(9~18시 KST): 300건
  - 저녁(19~23시): 180건
  - 새벽: 60건
- **7일 누적 예상**: ~8,400건 → Nikto 450건과 혼합 시 95:5 비율 달성
- **수동 트리거 결과**: 300건 성공, 실패 0건 확인

- `scripts/` 폴더 없음, `train_model.py` / `federated_learning.py` 없음

**결론**: Lambda 구조와 데이터는 그룹 프로젝트에서 가져왔고,
