# [260513] 보안 관제 대시보드 구성 — 1단계 AI 메트릭 + 2단계 WAF 로그 Athena 연동

## 1. 배경

AI 탐지 엔진 재학습(260512) 이후, 재작업 결과를 실제로 시각화하는 작업이 남아 있었다.  
`monitor.py`가 Prometheus에 `predicted_risk`, `observed_risk`, `anomaly_status` 등을 이미 노출하고 있었지만, Isolation Forest의 핵심 출력값인 `raw_score`(원점수)와 점수 분포가 빠져 있었다.

또한 WAF 원본 로그 기반 보안 관제 지표(룰별 차단 건수, 공격 유형 분포, 국가별 트래픽)를 대시보드에 추가하기 위해 S3 WAF 로그 상태 확인과 Athena 연동 작업이 필요했다.

## 3. 1단계 — monitor.py Prometheus 메트릭 추가

### 추가한 메트릭

| 메트릭명 | 타입 | 설명 |
|---|---|---|
| `aiops_raw_score` | Gauge | Isolation Forest decision_function 원점수 (양수=정상, 음수=이상) |
| `aiops_anomaly_threshold` | Gauge | 탐지 경계값 (전체 점수의 5th percentile, 고정값) |
| `aiops_score_distribution` | Histogram | 점수 분포 버킷 (IQR 0.07 기준 0.05 간격 설계) |

### 수정 내용 (`ai/inference/monitor.py`)

- `prometheus_client` import에 `Histogram` 추가
- 메트릭 선언부에 3개 메트릭 추가
- 모델 로드 직후 `ANOMALY_THRESHOLD.set(THRESHOLD)` 1회 등록
- 메인 루프에서 매 틱마다 `RAW_SCORE.set()`, `SCORE_HIST.observe()` 기록

### Grafana 패널 구성 계획

| 패널 | 쿼리 | 설명 |
|---|---|---|
| AI Score 추이 | `aiops_raw_score`, `aiops_anomaly_threshold` | threshold 수평선 기준으로 정상/이상 구간 확인 |
| Anomaly Status | `aiops_anomaly_status` | 0=정상(초록) / 1=이상(빨강) Stat 패널 |
| 예측/실측 비교 | `aiops_predicted_risk`, `aiops_observed_risk` | 선행 탐지 효과 시각화 |
| Score 분포 | `aiops_score_distribution_bucket` | 경계값 기준 정상/이상 누적 분포 |

## 4. 2단계 — WAF 로그 S3 확인 및 Athena DDL 작성

### S3 로그 상태 확인

- 버킷: `aws-waf-logs-minju-0417-project`
- 로그 존재 기간: 2026-05-06 ~ 2026-05-11 (이후 트래픽 없음)
- 로그가 안 쌓이는 이유: Terraform 로깅 필터가 `BLOCK` 이벤트만 저장하는데, 현재 WAF에 실제 트래픽이 없음

### 실제 경로 구조

```
s3://aws-waf-logs-minju-0417-project/
  AWSLogs/095035153545/WAFLogs/us-east-1/devsecops-waf/
    2026/{month}/{day}/{hour}/{minute}/
      {account}_waflogs_{region}_{webacl}_{timestamp}_{uuid}.log.gz
```

당초 hour까지만 예상했으나 실제로 minute 레벨 폴더가 하나 더 있음.  
Athena partition projection 템플릿을 `/${hour}/`까지로 지정하면 minute 하위 폴더를 재귀 스캔하므로 DDL 수정 없이 동작함.

### Athena DDL 핵심 설계 결정

- Partition Projection 적용 → `MSCK REPAIR TABLE` 불필요, 신규 로그 자동 인식
- `'has_encrypted_data' = 'true'` → KMS 암호화 버킷 필수 설정
- `JsonSerDe` + `ignore.malformed.json = true` → gzip JSON 파싱, 오류 행 무시
- DDL 파일 저장 위치: `docs/etc/athena_waf_ddl.sql`

### Athena 쿼리 검증 결과

```sql
SELECT event_time, action, terminatingruleid, client_ip, country, uri
FROM monitoring_db.waf_access_logs
WHERE year = '2026' AND month = '05' AND day = '06'
LIMIT 10;
```

실행 결과 정상 데이터 확인:
- 전체가 `GeoBlock-Non-KR` 룰로 차단된 비-KR IP (US, NL 등)
- NL IP `204.76.203.206` 반복 등장 → 봇/스캐너 추정
- `/autodiscover/autodiscover.json` URI → Exchange 서버 탐색 봇 패턴

## 5. Grafana 호스팅 아키텍처 결정

### 배경

대시보드 구성을 위해 Grafana를 어디에 올릴지 결정이 필요했다.  
초기에는 EC2 t3.micro + Docker + Nginx 리버스 프록시 방안을 검토했으나, 기각했다.

### 결정: Grafana Cloud (Managed SaaS)

**기각 이유 (EC2)**:  
이 프로젝트의 핵심 파이프라인은 Lambda/API Gateway 기반 서버리스 구조다.  
EC2를 추가하면 "서버리스 전환 완료"라는 아키텍처 메시지와 충돌하고, 면접에서 "EC2가 왜 있나요?" 질문이 생긴다.

**채택 이유 (Grafana Cloud)**:
- 서버리스 아키텍처 일관성 유지
- Grafana는 대시보드 설정과 세션을 저장하는 Stateful 서비스 → 핵심 파이프라인과 계층 분리가 자연스러움
- EC2 운영 부담 및 추가 비용 없음
- Prometheus + Athena 데이터소스 통합 수집 가능

**포트폴리오 설명 문구**:
> "핵심 보안 자동화는 AWS 서버리스로 구현하고, 상태 저장이 필요한 관제 UI는 Managed Observability SaaS로 분리했다."

### 코드 변경

- `infra/grafana.tf` 작성 후 삭제 (EC2 방식 기각)
- `variables.tf`에서 `grafana_key_pair_name`, `grafana_admin_password` 제거
- `README.md` 7절에 설계 의도 문서화

## 8. [260514] Grafana CloudWatch 대시보드 구성

### 아키텍처 재결정

Grafana Cloud + Athena 연결 시도 중 문제 확인: Athena 데이터소스는 IAM Access Key가 필요한데, 현재 프로젝트는 GitHub Actions OIDC 기반으로 장기 자격증명을 제거한 구조다. Athena 때문에 Access Key를 추가하면 포트폴리오 메시지가 흐려진다고 판단, 운영 메트릭을 CloudWatch 중심으로 전환했다.

**최종 Observability 3계층 구조**:

```
[로컬 / 데모 계층]   monitor.py → Prometheus metrics (:8000)
[운영 계층]          WAF / Lambda / API Gateway → CloudWatch Metrics & Logs
[시각화 계층]         Grafana Cloud (devsecai.grafana.net) → CloudWatch datasource
```

**Prometheus 유지 결정**: monitor.py의 Prometheus 메트릭은 삭제하지 않았다. "Prometheus를 구현해봤지만 서버리스 환경에서는 pull 방식이 맞지 않아 CloudWatch로 전환했다"는 설명이 판단력을 보여준다.

**IAM user 예외**: `grafana-cloudwatch-readonly` (CloudWatchReadOnlyAccess) 생성. Terraform 관리 제외 — state 파일 보안 리스크 때문.

### 완료된 패널

| 패널 | Namespace | Metric | Visualization |
|---|---|---|---|
| Real-time WAF Attack Blocking Trend | `AWS/WAFV2` | `BlockedRequests` (Rule=ALL) | Time series |
| WAF Blocked Requests by Rule | `AWS/WAFV2` | `BlockedRequests` (룰 4개 분리) | Bar chart |

룰별 쿼리 분리: `awsCommonRules`, `awsReputationRules`, `awsSQLiRules`, `geoBlockNonKR`  
`geoBlockNonKR` 31건 차단 확인.
